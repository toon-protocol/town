## Story

As a **skill author**,
I want a pipeline skill that converts any Nostr NIP into a TOON-aware Claude Agent Skill,
So that future NIPs can be converted to TOON skills without re-scoping the epic.

**Dependencies:** Stories 9.0 (social intelligence patterns -- done), 9.1 (TOON protocol context -- done)

**Decision sources:**
- Party Mode 2026-03-22: NIP Skills Epic (D9-001, D9-002, D9-003, D9-007, D9-009, D9-010)
- `_bmad-output/project-context.md` section "NIP-to-TOON Skill Pipeline Architecture"
- `_bmad-output/planning-artifacts/epics.md` Epic 9, Story 9.2
- `_bmad-output/planning-artifacts/test-design-epic-9.md` Story 9.2 (highest-risk story in epic)

**Downstream dependencies:** This is the **single point of failure** for ~30 downstream skills (Stories 9.4-9.33). Every NIP skill produced in Phases 1-10 is generated by this pipeline. Story 9.3 (eval framework) runs TOON compliance assertions that this pipeline auto-injects. Story 9.34 (publication gate) validates all pipeline-produced skills.

**Nature of deliverable:** This story produces a **Claude Agent Skill** (structured markdown + reference files + eval JSON + validation script), NOT TypeScript code. The output is a `.claude/skills/nip-to-toon-skill/` directory following Anthropic's skill-creator format. The skill teaches an agent HOW to convert any NIP into a TOON-compliant skill by following a 13-step pipeline.

**Risk context:** This is the **highest-risk story in Epic 9** (E9-R001, score 9/9). A defect in the pipeline propagates to every downstream skill. The pipeline has 14 acceptance criteria -- the most complex story in the epic. Meta-eval validation (running the pipeline on 3 test NIPs) is the primary mitigation.

**Rationale:** D9-001 chose "pipeline over catalog" -- build a skill factory, not just a skill collection. The `nip-to-toon-skill` pipeline enables future NIP conversion without re-scoping. The 30 NIP skills (Phases 1-10) are the first batch through this pipeline. The pipeline's lasting value is that when NIP-XX lands in the future, any agent with this skill can produce a TOON-compliant skill for it.

**Then** it spawns parallel subagents: one with the skill loaded, one without (baseline). Results saved to `with_skill/outputs/` and `without_skill/outputs/`.

### AC10: Grading + Benchmarking Steps [Test: 9.2-STEP-008]
**Given** the pipeline Steps 9-10 (Grading + Benchmarking)

**Then** it runs TOON-specific assertion checks -- red = skill is not TOON-ready.

### SKILL.md Format Requirements (from skill-creator)

- **Frontmatter:** YAML with ONLY `name` and `description` fields. No other frontmatter fields (`license`, `version`, `author` etc. are forbidden).
- **Description is the trigger mechanism.** Claude reads ONLY `name` + `description` to decide if the skill activates. All "when to use" information must be in the description, not the body. Target ~80-120 words.
- **Body:** Loaded only after skill triggers. Keep under 500 lines / ~5k tokens. Use imperative/infinitive form.
- **No extraneous files:** No README.md, INSTALLATION_GUIDE.md, CHANGELOG.md, QUICK_REFERENCE.md, etc.
- **References are loaded on-demand:** Claude reads reference files only when it determines they are needed. SKILL.md body must describe when to read each reference file.
- **Progressive disclosure:** Level 1 = frontmatter (~100 tokens, always loaded). Level 2 = SKILL.md body (<5k tokens, loaded on trigger). Level 3 = references (unlimited, loaded as-needed).

### Pipeline Architecture (from epics.md and project-context.md)

The 13-step pipeline converts any NIP into a TOON-aware skill:

1. **NIP Analysis** -- Read NIP spec. Classify as read-only / write-capable / both. Identify event kinds, tag structures, content formats.
2. **TOON Context Injection** -- Inject TOON write model (for write-capable) or TOON read model (for read-capable) from `toon-protocol-context.md`. All skills get relay discovery context.
3. **Social Context Layer** -- Generate `## Social Context` using `social-context-template.md`. Must be NIP-specific, not generic.
4. **Skill Authoring** -- Generate SKILL.md: frontmatter, body <5k tokens, Level 3 references. Follow skill-creator anatomy.
5. **Eval Generation** -- Generate `evals/evals.json`: 8-10 trigger queries, 8-10 non-trigger queries, 4-6 output evals with assertions.
6. **TOON Assertions** -- Auto-inject the 5 TOON compliance assertions based on NIP classification.
7. **Description Optimization** -- Run `scripts.run_loop` with 20 trigger queries, max 5 iterations.
8. **With/Without Testing** -- Parallel subagent runs: with-skill vs baseline Claude.
9. **Grading** -- Produce `grading.json` with per-assertion results.
10. **Benchmarking** -- Produce `benchmark.json` with pass rate, timing, token usage.
11. **TOON Compliance Validation** -- Run TOON assertion checks. Red = skill not ready.
12. **Eval Viewer** -- Generate HTML review via `eval-viewer/generate_review.py`.
13. **Iterate** -- Read feedback, refine, re-run into `iteration-2+/`.

### NIP Classification Rules

Classification drives which TOON context sections are injected:

| Classification | Criteria | TOON Context Injected |
|---------------|----------|----------------------|
| **Read-only** | NIP defines filters/queries but no new event kinds for writing (e.g., NIP-50 Search) | `toon-format-check`, TOON read model |
| **Write-capable** | NIP defines event kinds the agent creates/publishes (e.g., NIP-25 Reactions) | `toon-write-check`, `toon-fee-check`, TOON write model + fee calculation |
| **Both** | NIP defines both queryable data and publishable event kinds (e.g., NIP-23 Long-form) | All checks: write + read + fee |
| **All** | Universal checks applied regardless of classification | `social-context-check`, `trigger-coverage` |

### TOON Compliance Assertion Templates

These 5 assertions are auto-injected by the pipeline (Step 6):

1. **`toon-write-check`** (write-capable only): Skill uses `publishEvent()` from `@toon-protocol/client`, NOT bare `["EVENT", ...]`. References `nostr-protocol-core` for write model details.
2. **`toon-fee-check`** (write-capable only): Skill includes fee calculation or references `fee-calculation.md`. Formula: `basePricePerByte * serialized event bytes`.
3. **`toon-format-check`** (read-capable only): Skill documents that TOON relay returns TOON-format strings, not JSON objects.
4. **`social-context-check`** (all): Skill has `## Social Context` section that is NIP-specific (not a generic placeholder). References `nostr-social-intelligence` for deeper social guidance.
5. **`trigger-coverage`** (all): Skill description includes social-situation triggers, not just protocol-technical triggers.

### `toon-protocol-context.md` -- The Canonical Reference (D9-010)

This file is the single source of truth for TOON protocol details. It was first authored in Story 9.1 at `.claude/skills/nostr-protocol-core/references/toon-protocol-context.md`. This pipeline skill has its own copy because it injects this content into every generated skill. Both copies MUST be consistent. When the protocol changes, update the `nostr-protocol-core` version first, then propagate to this pipeline skill.

Key content:
- TOON write model: `client.publishEvent(event, { destination, amount?, bid? })` -- no condition/fulfillment (D9-005)
- TOON read model: standard NIP-01 subscriptions, relay returns TOON-format strings
- Transport: `@toon-protocol/client` (NOT the SDK)
- Relay discovery: enriched NIP-11 `/health` endpoint, kind:10032 pricing events
- Social economics: ILP paid-writes create quality floor, cost-per-byte shapes norms

### `validate-skill.sh` -- Skill Linting Script

This script validates any pipeline-generated skill. It is used by:
- This story: self-validation of the pipeline skill itself
- Story 9.3 (eval framework): automated validation in batch runner
- Stories 9.4-9.33: validation of each generated skill
- Story 9.34 (publication gate): final sweep

Implementation notes:
- Input: path to skill directory (e.g., `.claude/skills/nostr-social-interactions/`)
- Checks: (1) SKILL.md exists, (2) YAML frontmatter valid, (3) `references/` exists, (4) `evals/evals.json` valid JSON, (5) `## Social Context` present, (6) no bare `["EVENT", ...]`, (7) description word count 50-200, (8) body under 500 lines
- Output: pass/fail with per-check results
- Exit code: 0 = all checks pass, 1 = at least one check failed
- Dependencies: bash, grep, wc, node (for JSON validation). No npm dependencies.

### Anti-Patterns to Avoid (Dev Agent Guardrails)

- **DO NOT create README.md, CHANGELOG.md, or any file not in the Output Directory tree above.** The skill-creator explicitly forbids extraneous documentation files.
- **DO NOT add frontmatter fields beyond `name` and `description`.** No `license`, `version`, `author`, `tags`, etc.
- **DO NOT put "when to use" guidance in the body.** The body is loaded AFTER triggering. All trigger information goes in the `description` field.
- **DO NOT write bare `["EVENT", ...]` patterns anywhere.** Always reference `publishEvent()` from `@toon-protocol/client` for writes.
- **DO NOT duplicate content between SKILL.md body and reference files.** The body should point to references, not repeat their content.
- **DO NOT use `scripts/init_skill.py` or `scripts/package_skill.py`.** These are skill-creator tools for generic skill creation. This story creates files directly since the structure is fully specified.
- **DO NOT write a generic `## Social Context` template.** The template must produce NIP-specific social context. Test: could this Social Context section apply to ANY NIP? If yes, it is too generic.
- **DO NOT conflate the pipeline skill with pipeline-GENERATED skills.** This story creates the pipeline skill (the factory). Pipeline-generated skills are the output when an agent uses this skill.
- **DO NOT confuse the SDK with the client.** The SDK (`createNode()`, `HandlerRegistry`) is for service node PROVIDERS. The client (`publishEvent()`) is for agents sending events. Pipeline-generated skills teach agents, so they use the client.

### Test Strategy

- **Structural validation:** Verify all files exist, frontmatter is valid YAML with only `name` + `description`, body under 500 lines, no extraneous files, no bare `["EVENT", ...]` patterns. - **Pipeline step validation:** Verify each of the 13 pipeline steps is documented with actionable instructions. Tests: 9.2-STEP-001 through 9.2-STEP-009.
- **Meta-eval validation (CRITICAL):** Run the pipeline mentally on 3 test NIPs to verify it produces correct output: (1) NIP-50 Search (read-only), (2) NIP-25 Reactions (write-capable), (3) NIP-23 Long-form (read+write). Tests: 9.2-META-001 through 9.2-META-004.
